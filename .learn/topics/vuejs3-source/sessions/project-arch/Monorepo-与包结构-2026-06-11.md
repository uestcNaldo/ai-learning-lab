## 定位

`Monorepo 与包结构` 是 Vue 3 源码学习的入口地图：它告诉你每个 `packages/*` 包负责什么、谁依赖谁、以及后面读响应式、渲染器、组件系统和编译器时应该从哪里进。

## 类比

可以把 Vue 3 的 monorepo 想成一座大型剧院。`@vue/reactivity` 是后台的感应系统，知道哪些道具变了；`@vue/runtime-core` 是舞台调度中心，决定组件怎么挂载、更新和卸载；`@vue/runtime-dom` 是面向浏览器舞台的执行团队，负责真实 DOM 操作；`@vue/compiler-*` 是剧本改编组，把模板改写成 render function；`vue` 主包则像前台售票口，把常用能力打包给应用开发者。

## 核心机制

Vue 3 不是一个单文件框架，而是一个由多个 package 组合起来的 monorepo。这样设计的好处是：每个包有清晰职责，可以被单独构建、测试、发布，也能让不同平台复用同一套核心逻辑。

源码学习时最重要的是先看懂这条分层关系：

1. `@vue/shared` 提供跨包工具函数和常量，例如类型判断、字符串处理、全局编译标记等。它像地基，很多包都会依赖它。
2. `@vue/reactivity` 实现响应式系统，包括 `reactive`、`ref`、`effect`、`computed` 等。它可以独立于 DOM 使用。
3. `@vue/runtime-core` 实现平台无关的运行时核心，包括 VNode、组件实例、组件更新、调度器、生命周期、renderer 抽象等。
4. `@vue/runtime-dom` 基于 `runtime-core` 提供浏览器 DOM 平台能力，例如 `createElement`、`patchProp`、事件处理、DOM 属性更新，并导出面向浏览器应用的 `createApp`。
5. `@vue/compiler-core` 提供平台无关的模板编译核心，包括 AST、transform、codegen 等。
6. `@vue/compiler-dom` 在 `compiler-core` 之上增加 DOM 平台的编译规则，例如 DOM 属性、事件、指令处理。
7. `@vue/compiler-sfc` 处理 `.vue` 单文件组件，把 `<script>`、`<template>`、`<style>` 拆开并交给不同编译流程。官方资料中也强调，现代 Vue 版本里通常通过 `vue/compiler-sfc` 这个 deep import 使用它。
8. `vue` 主包整合 runtime 和 compiler 相关能力，对普通应用暴露最常用的入口。

读源码时不要把这些包看成平铺目录，而要看成一条数据流：

`template` → `compiler` 生成 `render function` → `render function` 返回 `VNode` → `runtime-core` patch VNode → `runtime-dom` 操作真实 DOM → `reactivity` 触发重新渲染。

也可以从运行时反过来看：

状态变化先进入 `reactivity`，触发组件的 render effect；组件重新执行 render function 得到新 VNode；renderer 比较新旧 VNode；最后由 `runtime-dom` 把差异应用到浏览器 DOM。

这就是为什么源码学习要先建立包结构地图。没有地图时，你会在 `runtime-core`、`runtime-dom`、`compiler-core`、`compiler-dom` 之间迷路；有地图后，每个文件都能放回它该在的位置。

## 代码示例

```ts
// A tiny mental model of Vue 3 package relationships.
// This is not Vue's real source code; it is a reading map.

type PackageNode = {
  role: string
  dependsOn: string[]
}

const vuePackages: Record<string, PackageNode> = {
  "@vue/shared": {
    role: "跨包共享工具和常量",
    dependsOn: [],
  },
  "@vue/reactivity": {
    role: "响应式系统：reactive/ref/effect/computed",
    dependsOn: ["@vue/shared"],
  },
  "@vue/runtime-core": {
    role: "平台无关运行时：VNode、组件、renderer 抽象、调度",
    dependsOn: ["@vue/shared", "@vue/reactivity"],
  },
  "@vue/runtime-dom": {
    role: "浏览器 DOM 运行时：DOM 操作、事件、属性 patch",
    dependsOn: ["@vue/runtime-core", "@vue/shared"],
  },
  "@vue/compiler-core": {
    role: "平台无关模板编译：parse、transform、codegen",
    dependsOn: ["@vue/shared"],
  },
  "@vue/compiler-dom": {
    role: "DOM 平台模板编译规则",
    dependsOn: ["@vue/compiler-core", "@vue/shared"],
  },
  "@vue/compiler-sfc": {
    role: ".vue 单文件组件编译",
    dependsOn: ["@vue/compiler-core", "@vue/compiler-dom", "@vue/shared"],
  },
  "vue": {
    role: "面向应用开发者的主入口",
    dependsOn: ["@vue/runtime-dom", "@vue/compiler-dom", "@vue/compiler-sfc"],
  },
}

function explainPackage(name: keyof typeof vuePackages) {
  const pkg = vuePackages[name]
  return `${name}: ${pkg.role}; depends on: ${pkg.dependsOn.join(", ") || "none"}`
}

console.log(explainPackage("@vue/runtime-dom"))
```

这段代码不是 Vue 的真实源码，而是一张可执行的脑内地图。你可以从 `@vue/runtime-dom` 看出一个关键事实：浏览器应用看起来是在用 DOM API，但它真正站在 `runtime-core` 之上；而 `runtime-core` 又复用了 `reactivity`。这解释了为什么 Vue 可以有自定义 renderer：只要替换平台操作，核心组件更新逻辑仍然可以复用。

## 常见误区

1. 误以为 `vue` 主包就是全部源码。实际学习时，主包更像入口聚合层，很多核心实现分散在 `reactivity`、`runtime-core`、`runtime-dom`、`compiler-*` 中。
2. 误以为 `runtime-core` 会直接操作 DOM。它是平台无关层，真正的 DOM 操作在 `runtime-dom` 里。
3. 误以为响应式系统只服务组件。`@vue/reactivity` 可以独立使用，组件更新只是它的一个重要应用场景。
4. 误以为编译器和运行时可以完全分开理解。模板编译最终服务于 runtime patch，所以理解 VNode、PatchFlags、renderer 后再看 compiler 会更顺。
5. 误以为一开始就应该钻 `diff`。如果还不知道 VNode、renderer、组件 render effect 在哪里，直接读 diff 很容易变成逐行硬啃。

## 思考检查

1. 如果 `@vue/runtime-core` 是平台无关的，那浏览器 DOM 相关的 `createElement`、事件绑定、属性更新应该放在哪个包里？答案是 `@vue/runtime-dom`。
2. 为什么建议先看 `@vue/shared` 和包结构，再看 `reactivity`？因为源码里跨包工具和依赖方向会反复出现，先知道地基能减少后面读代码时的噪音。

---

## 快速总结
- Vue 3 monorepo 的核心不是目录很多，而是职责分层清楚：shared、reactivity、runtime、compiler、主入口各管一段。
- 源码主链路可以理解为：compiler 生成 render function，runtime 执行并 patch VNode，runtime-dom 落到真实 DOM，reactivity 触发更新。
- 学源码先建立包结构地图，后面读 `reactive`、`effect`、`renderer`、`compiler` 时才知道文件属于哪一层。
