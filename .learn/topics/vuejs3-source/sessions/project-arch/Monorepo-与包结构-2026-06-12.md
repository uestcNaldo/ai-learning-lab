## Positioning

`Monorepo 与包结构` 是进入 Vue 3 源码的地图层：它帮助你把每个 `packages/*` 包放到正确职责位置，先知道“谁负责什么”，再去读具体实现。

## Analogy

可以把 Vue 3 想成一个剧组。`shared` 是所有人都会用的工具箱；`reactivity` 是感应系统，负责发现状态变化；`runtime-core` 是总导演，安排组件、VNode、更新调度和 renderer 抽象；`runtime-dom` 是浏览器舞台组，把导演的安排落到真实 DOM；`compiler-*` 是剧本改编组，把模板改成 render function；`vue` 主包是对外售票口，把常用能力整理成用户最常 import 的入口。

## Core Mechanism

Vue 3 的源码仓库是 monorepo，核心代码分布在多个 package 中。这个设计的关键不是“目录多”，而是“边界清楚”：响应式可以独立于 DOM，运行时核心可以独立于具体平台，DOM 渲染器只负责浏览器平台，编译器可以把模板转换成 runtime 能消费的 render function。

下面是当前练习最需要掌握的 8 个包。

### `@vue/shared`

`@vue/shared` 是跨包共享工具包。它不承载某个业务主线，而是提供很多底层工具、常量和类型判断，例如 `NOOP`、`extend`、`isObject`、`isString`、`hasChanged`、`camelize`、`hyphenate`、`PatchFlags`、`ShapeFlags` 等。你可以把它看成 Vue 内部的标准工具箱。

它几乎不会依赖别的 Vue 包，反过来会被 `reactivity`、`runtime-core`、`runtime-dom`、`compiler-core`、`compiler-dom`、`compiler-sfc` 等大量包依赖。读源码时，如果看到一个函数来自 `@vue/shared`，通常不要先深挖它，只要知道它是跨包复用的基础工具。

### `@vue/reactivity`

`@vue/reactivity` 是响应式系统包，负责 `reactive`、`ref`、`effect`、`computed`、`watch`、`track`、`trigger` 等能力。它解决的问题是：数据变化后，哪些副作用需要重新执行？

它依赖 `@vue/shared`，但不依赖 `runtime-dom`，所以它不需要浏览器 DOM 也能工作。官方 README 也说明它可以作为独立包发布，同时用户侧 renderer 会重新导出这些响应式 API。这个边界很重要：`reactive` 属于数据层，不属于浏览器渲染层。

### `@vue/runtime-core`

`@vue/runtime-core` 是平台无关运行时核心。它负责 VNode、组件实例、组件挂载和更新、生命周期、调度器、依赖响应式的 render effect，以及 `createRenderer` 这类 renderer 抽象。

它依赖 `@vue/shared` 和 `@vue/reactivity`。这很好理解：运行时既需要工具函数，也需要响应式系统来驱动组件更新。但它仍然不直接操作 DOM。官方 README 说它主要用于 typing 和构建自定义 renderer，不是给普通应用直接使用。`createRenderer` 接收平台操作，比如 `insert`、`remove`、`createElement`、`patchProp`，然后生成 `render` 和 `createApp`。

所以 `h`、`createVNode`、`createRenderer` 这类 API 更偏 `runtime-core`，因为它们描述的是平台无关的渲染模型。

### `@vue/runtime-dom`

`@vue/runtime-dom` 是浏览器 DOM 平台运行时。它在 `runtime-core` 的 renderer 抽象之上，补上浏览器需要的具体操作：`nodeOps`、`patchProp`、DOM 事件、DOM 属性、`Transition`、`vShow`、`vModel` 等。

它的 `src/index.ts` 会通过 `createRenderer(rendererOptions)` 创建浏览器 renderer，然后导出浏览器版 `render`、`hydrate`、`createApp`、`createSSRApp`。它最后还会 `export * from '@vue/runtime-core'`，所以从 `@vue/runtime-dom` 也能拿到 `h`、组件 API、响应式 API 等。

练习里把 `createApp` 归到 `@vue/runtime-dom`，是因为普通浏览器应用的 `createApp(...).mount('#app')` 需要 DOM container 解析、清空挂载点、设置 `data-v-app` 等浏览器平台行为。

### `@vue/compiler-core`

`@vue/compiler-core` 是平台无关模板编译核心。它负责通用的 `baseParse`、`transform`、`generate`、AST 类型、编译错误、基础 transform 等。

它不应该直接绑定浏览器 DOM 规则。你可以把它理解成“模板编译的通用发动机”：把模板字符串变成 AST，遍历和转换 AST，再生成 render function 代码。练习里 `parse` 归到 `@vue/compiler-core`，指的是通用 parser 能力。

### `@vue/compiler-dom`

`@vue/compiler-dom` 是 DOM 平台模板编译器。它基于 `compiler-core`，加入 DOM 平台规则，例如 DOM parser options、`v-html`、`v-text`、`v-model`、`v-on`、`v-show`、HTML 嵌套校验、静态字符串化等。

它依赖 `@vue/compiler-core` 和 `@vue/shared`。它的核心价值是：让通用编译器知道浏览器 DOM 模板有哪些特殊规则。比如同样是 `v-model`，DOM 上的 input/select/checkbox/radio 有平台细节，这就不该放在 `compiler-core` 里。

### `@vue/compiler-sfc`

`@vue/compiler-sfc` 是 `.vue` 单文件组件编译工具包。它处理的不是单纯 template，而是整个 SFC 文件：`<template>`、`<script>`、`<script setup>`、`<style>`、`<style scoped>`、CSS modules、CSS vars 等。

它会导出 `parse`、`compileTemplate`、`compileScript`、`compileStyle` 等能力。官方 README 说明它是给 bundler/plugin/loader 使用的低层工具，比如 `@vitejs/plugin-vue` 和 `vue-loader`。所以练习里 `compileTemplate` 归到 `@vue/compiler-sfc`，因为这是 SFC 编译流程里的模板编译入口。

### `vue`

`vue` 是面向应用开发者的主入口包。它不是“所有实现都写在这里”，而是聚合其他包，对外提供最常用的 import 入口。

在 `packages/vue/src/index.ts` 中，full build 会引入 `@vue/compiler-dom` 和 `@vue/runtime-dom`，注册 runtime compiler，并最终 `export * from '@vue/runtime-dom'`。这解释了为什么普通项目里常写 `import { createApp, ref } from 'vue'`：`vue` 把 runtime-dom、runtime-core、reactivity 等能力整理后暴露给用户。

## Code Example

```ts
type PackageRole = {
  layer: string
  owns: string[]
  dependsOn: string[]
}

const vuePackageRoles: Record<string, PackageRole> = {
  "@vue/shared": {
    layer: "基础工具层",
    owns: ["NOOP", "extend", "isObject", "PatchFlags", "ShapeFlags"],
    dependsOn: [],
  },
  "@vue/reactivity": {
    layer: "响应式层",
    owns: ["reactive", "ref", "effect", "computed", "track", "trigger"],
    dependsOn: ["@vue/shared"],
  },
  "@vue/runtime-core": {
    layer: "平台无关运行时层",
    owns: ["h", "createVNode", "component instance", "scheduler", "createRenderer"],
    dependsOn: ["@vue/shared", "@vue/reactivity"],
  },
  "@vue/runtime-dom": {
    layer: "浏览器 DOM 运行时层",
    owns: ["createApp", "render", "hydrate", "nodeOps", "patchProp"],
    dependsOn: ["@vue/runtime-core", "@vue/shared"],
  },
  "@vue/compiler-core": {
    layer: "平台无关编译核心",
    owns: ["baseParse", "transform", "generate", "AST"],
    dependsOn: ["@vue/shared"],
  },
  "@vue/compiler-dom": {
    layer: "DOM 平台编译层",
    owns: ["compile", "parse", "DOMDirectiveTransforms", "parserOptions"],
    dependsOn: ["@vue/compiler-core", "@vue/shared"],
  },
  "@vue/compiler-sfc": {
    layer: "SFC 编译层",
    owns: ["parse", "compileTemplate", "compileScript", "compileStyle"],
    dependsOn: ["@vue/compiler-core", "@vue/compiler-dom", "@vue/shared"],
  },
  vue: {
    layer: "用户入口聚合层",
    owns: ["createApp re-export", "ref re-export", "runtime compiler full build"],
    dependsOn: ["@vue/runtime-dom", "@vue/compiler-dom", "@vue/compiler-sfc"],
  },
}

function findOwner(api: string) {
  return Object.entries(vuePackageRoles).find(([, info]) =>
    info.owns.some(name => name.includes(api)),
  )?.[0]
}

console.log(findOwner("reactive")) // @vue/reactivity
console.log(findOwner("createRenderer")) // @vue/runtime-core
console.log(findOwner("createApp")) // @vue/runtime-dom 或 vue 的 re-export
```

这段代码是一张脑内地图。注意 `createApp` 的判断有一点细：从用户角度常从 `vue` import，但实现和浏览器平台行为来自 `@vue/runtime-dom`；而 `runtime-dom` 又基于 `runtime-core` 的 `createRenderer`。

## Common Misconceptions

1. 误以为 `vue` 主包包含所有实现。实际它更多是用户入口和聚合层，核心实现分布在 runtime、reactivity、compiler 等包里。
2. 误以为 `runtime-core` 会操作 DOM。它只定义平台无关的 renderer 机制，DOM 操作由 `runtime-dom` 提供。
3. 误以为 `reactive` 属于组件系统。响应式包可以独立使用，组件系统只是把它用于 render effect。
4. 误以为 `compiler-core` 就能处理所有 DOM 模板细节。DOM 指令和平台规则属于 `compiler-dom`。
5. 误以为 `compiler-sfc` 只是编译 template。它处理整个 `.vue` 文件，包括 script、template、style 和构建工具集成。

## Socratic Check

1. 如果要做一个 Canvas renderer，你会复用 `runtime-core` 还是 `runtime-dom`？答案是复用 `runtime-core`，然后自己提供 Canvas 平台的 `insert`、`remove`、`createElement` 等等价操作。
2. 如果你看到 `compileTemplate`，它更像是 `compiler-core`、`compiler-dom` 还是 `compiler-sfc` 的职责？答案是 `compiler-sfc`，因为它处在 `.vue` 单文件组件编译流程里。

---

## Quick Summary
- `shared -> reactivity -> runtime-core -> runtime-dom` 是理解 Vue runtime 侧的主依赖链。
- `compiler-core -> compiler-dom -> compiler-sfc` 是理解 Vue compiler 侧的分层链路。
- `vue` 主包是面向用户的聚合入口，不等于所有核心实现都在 `packages/vue` 里。
