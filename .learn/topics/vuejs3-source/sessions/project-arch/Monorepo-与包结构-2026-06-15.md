## Positioning

这次聚焦 Vue 3 monorepo 里的主入口包 `vue`，理解它如何把 runtime、compiler、SFC compiler 和 SSR 能力聚合成用户安装的 `vue` 包。

## Analogy

如果把 Vue monorepo 看成一家汽车集团，`@vue/reactivity`、`@vue/runtime-core`、`@vue/runtime-dom`、`@vue/compiler-dom` 等包是不同专业工厂；`vue` 包不是亲自生产每一个零件的工厂，而是最终卖给用户的整车品牌门店：它把底盘、发动机、电控、选装配置和说明书打包成一个用户能直接购买和使用的产品。

## Core Mechanism

从你本地源码 `/Users/naldomac/Projects/vuejs/core/packages/vue/package.json` 看，`vue` 包声明的 workspace 依赖是：

```text
vue
├── @vue/shared
├── @vue/compiler-dom
├── @vue/runtime-dom
├── @vue/compiler-sfc
└── @vue/server-renderer
```

这里要分清两层关系：

第一层是 package.json 的发布依赖。`vue` 发布到 npm 后，需要把这些子包作为依赖一起解析，因为用户可能从 `vue` 包访问不同入口：

- `vue` 默认入口：用户平时 `import { createApp, ref } from 'vue'` 的入口。
- `vue/compiler-sfc`：构建工具处理 `.vue` 单文件组件时使用。
- `vue/server-renderer`：SSR 场景使用。
- `vue/jsx-runtime`：JSX runtime 场景使用。
- `vue/dist/*`：不同构建产物入口，例如 runtime-only、full build、global build、browser build。

第二层是源码实际入口依赖。`packages/vue/src/index.ts` 是 full build 入口，它同时接入运行时和模板编译器：

- 从 `@vue/compiler-dom` 引入 `compile`，用于把 template 字符串编译成 render function。
- 从 `@vue/runtime-dom` 引入并重新导出 DOM runtime 能力。
- 从 `@vue/shared` 引入 `NOOP`、`extend`、`genCacheKey`、`generateCodeFrame`、`isString` 等工具。
- 调用 `registerRuntimeCompiler(compileToFunction)`，把运行时编译能力注册进 runtime。

`packages/vue/src/runtime.ts` 则是 runtime-only 入口，它只导出 `@vue/runtime-dom`，并提供一个开发期警告版 `compile()`。这和 Context7 查到的 Vue 官方文档一致：Vue 有 runtime-only build 和 full build；runtime-only 不包含浏览器端 compiler，适合模板已在构建阶段预编译的应用；full build 包含 compiler，支持浏览器内模板编译，但体积和运行时成本更高。

所以，`vue` 包的本质不是“实现所有功能”，而是“选择入口并聚合功能”：

```text
应用代码 import 'vue'
        │
        ▼
      vue 包
        │
        ├── runtime-only build ──> @vue/runtime-dom ──> @vue/runtime-core ──> @vue/reactivity ──> @vue/shared
        │
        └── full build ─────────> @vue/compiler-dom + @vue/runtime-dom + @vue/shared
```

再展开 `vue` 的关键依赖角色：

- `@vue/runtime-dom`：浏览器 DOM 平台运行时，提供 `createApp`、DOM renderer、DOM 指令和 DOM 事件处理等。
- `@vue/compiler-dom`：把面向 DOM 的 template 编译成 render function，full build 的运行时模板编译依赖它。
- `@vue/shared`：跨包共享工具与常量，避免每个包重复实现基础判断和工具函数。
- `@vue/compiler-sfc`：处理 `.vue` 单文件组件，主要供 Vite、webpack loader、Rollup plugin 等构建工具使用。
- `@vue/server-renderer`：服务端渲染入口，供 SSR 应用从 `vue/server-renderer` 使用。

## Code Example

```ts
// 这是学习用的简化依赖图，不是源码里的完整构建脚本。
const vuePackageDependencies = {
  vue: [
    "@vue/shared",
    "@vue/compiler-dom",
    "@vue/runtime-dom",
    "@vue/compiler-sfc",
    "@vue/server-renderer",
  ],
}

const runtimeOnlyEntry = {
  file: "packages/vue/src/runtime.ts",
  imports: ["@vue/runtime-dom"],
  behavior: "只导出浏览器运行时；compile() 在开发环境给出警告",
}

const fullBuildEntry = {
  file: "packages/vue/src/index.ts",
  imports: ["@vue/compiler-dom", "@vue/runtime-dom", "@vue/shared"],
  behavior: "注册 runtime compiler，让浏览器内 template 字符串可以被编译成 render function",
}
```

这段代码把 `vue` 的依赖拆成三块看：

1. `package.json` 层面，`vue` 声明依赖五个 workspace 包，因为它要对外暴露多个子入口。
2. runtime-only 入口层面，`vue` 主要就是 `@vue/runtime-dom` 的再导出。
3. full build 入口层面，`vue` 会额外连接 `@vue/compiler-dom`，把模板编译器注册到运行时。

如果你之后读源码，建议先从 `packages/vue/src/runtime.ts` 看 runtime-only，再读 `packages/vue/src/index.ts` 看 full build。顺序不要反过来；否则一上来会被 `compileToFunction`、缓存、错误 code frame、global build 分支绕晕。

## Common Misconceptions

1. 误以为 `vue` 包就是 Vue 的全部源码。真实情况是 `vue` 主包主要负责入口聚合，底层能力分散在 `runtime-*`、`compiler-*`、`reactivity`、`shared` 等包里。
2. 误以为 `vue` 一定依赖 compiler。更精确地说，`vue` package.json 声明了 `@vue/compiler-dom`，但实际应用常用的 bundler runtime-only 入口不会把浏览器端 compiler 带进最终产物。
3. 误以为 `@vue/compiler-sfc` 是浏览器运行时必需依赖。它主要服务构建阶段，处理 `.vue` 文件，不是普通浏览器运行时渲染必须携带的模块。
4. 误以为 `@vue/server-renderer` 会进入普通客户端 bundle。它是 `vue/server-renderer` 子入口依赖，主要服务 SSR。
5. 误以为 `@vue/runtime-dom` 只依赖 `@vue/runtime-core`。真实源码里它还会直接依赖 `@vue/shared`，并在部分类型或工具路径上接触 `@vue/reactivity`。

## Socratic Check

1. 如果一个项目用 Vite 构建 `.vue` 文件，为什么生产 bundle 通常不需要把 `@vue/compiler-dom` 的运行时编译能力带到浏览器？因为 template 已经在构建阶段被 `@vue/compiler-sfc` 和相关 compiler 包预编译成 render function。
2. 为什么 `vue` 包既声明 `@vue/compiler-sfc`，又声明 `@vue/runtime-dom`？因为它既要服务构建工具通过 `vue/compiler-sfc` 处理 SFC，也要服务应用运行时通过 `vue` 默认入口渲染 DOM。

## Quick Summary
- `vue` 是主入口聚合包，声明依赖 `@vue/shared`、`@vue/compiler-dom`、`@vue/runtime-dom`、`@vue/compiler-sfc`、`@vue/server-renderer`。
- runtime-only 入口主要再导出 `@vue/runtime-dom`；full build 入口额外接入 `@vue/compiler-dom` 并注册 runtime compiler。
- 读 `vue` 包依赖时要区分 package.json 发布依赖、源码入口实际 import、最终 bundle 是否真的包含某个模块。
