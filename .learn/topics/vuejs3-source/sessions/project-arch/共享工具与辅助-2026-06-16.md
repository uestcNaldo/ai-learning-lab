## Positioning

`共享工具与辅助` 位于 Vue 源码学习的工程架构层，是理解 `@vue/shared` 为什么被 `reactivity`、`runtime-core`、`runtime-dom`、`compiler-*` 等包反复依赖的基础概念。

## Analogy

可以把 `@vue/shared` 想成 Vue 源码仓库里的“公共工具间”。不同团队都要干活：响应式团队要判断对象和值是否变化，运行时团队要处理 class/style/props，编译器团队要生成 PatchFlags 和识别内置指令。与其每个团队自己造一套螺丝刀、卷尺和标签机，不如把这些稳定的小工具放进公共工具间，谁需要谁拿。

这个公共工具间有两个特点：

- 它自己尽量不依赖其他 Vue 包，避免形成循环依赖。
- 它服务多个方向，既有普通工具函数，也有编译器和运行时共同理解的“协议常量”。

## Core Mechanism

`@vue/shared` 的 package 描述是 `internal utils shared across @vue packages`。从本地 Vue 源码看，它的 `src/index.ts` 统一导出这些类别：

- 通用工具：`EMPTY_OBJ`、`EMPTY_ARR`、`NOOP`、`NO`、`extend`、`remove`、`hasOwn`。
- 类型判断：`isArray`、`isObject`、`isString`、`isFunction`、`isPromise`、`isPlainObject`、`toRawType`。
- 字符串和事件名转换：`camelize`、`hyphenate`、`capitalize`、`toHandlerKey`。
- 编译器和运行时协议：`PatchFlags`、`ShapeFlags`、`SlotFlags`。
- props/class/style 规范化：`normalizeClass`、`normalizeStyle`、`normalizeProps`。
- 编译/运行辅助：`makeMap`、`isBuiltInDirective`、`isReservedProp`、`genPropsAccessExp`。

它重要的原因不是“函数很多”，而是它降低了 Vue monorepo 内部包之间的耦合。比如：

`@vue/reactivity` 需要 `isObject`、`hasChanged` 这类基础判断，但它不应该依赖 runtime 或 compiler。

`@vue/runtime-dom` 需要 `normalizeClass`、`normalizeStyle` 处理用户传入的 class/style，但它不应该自己维护一套和 compiler 不一致的规则。

`@vue/compiler-dom` 会生成带 `PatchFlags` 的 render function，`@vue/runtime-core` 在 patch 阶段读取这些 flag 做优化。这里 `PatchFlags` 就像编译器和运行时之间的“暗号表”：编译器负责贴标签，运行时负责按标签走快路径。

`ShapeFlags` 则更像 VNode 的快速分类标签。一个 VNode 是普通元素、组件、文本子节点、数组子节点还是 slots 子节点，都可以通过位运算快速判断。这样运行时不用频繁做昂贵的结构判断。

所以读 Vue 源码时，可以把 `@vue/shared` 分成三层理解：

第一层是普通 JavaScript 工具：类型判断、对象操作、字符串转换。

第二层是输入规范化：把用户灵活写法整理成 runtime 更好处理的统一形状，例如 class 可以是字符串、数组、对象，最后都归一成字符串。

第三层是跨包协议：`PatchFlags`、`ShapeFlags` 这种常量不是普通工具，而是 compiler 和 runtime 之间共享的约定。

## Code Example

```ts
import {
  PatchFlags,
  ShapeFlags,
  hasChanged,
  isObject,
  normalizeClass,
  normalizeStyle,
} from '@vue/shared'

const classValue = normalizeClass([
  'btn',
  { active: true, disabled: false },
  ['primary'],
])

const styleValue = normalizeStyle([
  'color: red;',
  { fontSize: '14px' },
])

const shouldTrack = isObject({ count: 1 })
const shouldTrigger = hasChanged(2, 1)

const vnodeShape =
  ShapeFlags.ELEMENT |
  ShapeFlags.ARRAY_CHILDREN

const patchHint =
  PatchFlags.CLASS |
  PatchFlags.STYLE

console.log(classValue)
console.log(styleValue)
console.log(shouldTrack, shouldTrigger)
console.log(Boolean(vnodeShape & ShapeFlags.ELEMENT))
console.log(Boolean(patchHint & PatchFlags.CLASS))
```

这段代码展示了 `@vue/shared` 的三类作用。

第一，`normalizeClass` 和 `normalizeStyle` 把用户可以灵活书写的输入整理成 Vue 内部更统一的形式。用户可以写字符串、数组、对象，但 runtime 不应该每次都面对无限多形态。

第二，`isObject` 和 `hasChanged` 是响应式系统会反复使用的底层判断。响应式系统要知道一个值是否值得代理，也要知道新旧值是否真的变化。

第三，`ShapeFlags` 和 `PatchFlags` 是位掩码。它们允许 Vue 用非常便宜的位运算描述 VNode 类型和更新提示。比如 `patchHint & PatchFlags.CLASS` 为真，就表示这个节点的 class 是动态的，runtime 可以只走 class 更新相关逻辑。

## Common Misconceptions

误区一：把 `@vue/shared` 当成普通 utils 文件夹。它确实有很多 utils，但更关键的是它承载了跨包协议，比如 `PatchFlags` 和 `ShapeFlags`。

误区二：认为所有包都依赖 `@vue/shared` 只是为了省代码。真正原因是避免 runtime、compiler、reactivity 各自维护不同规则，导致行为不一致。

误区三：认为 `PatchFlags` 只属于 compiler。实际上 compiler 生成它，runtime 读取它；它是两边共享的优化协议。

误区四：认为 `@vue/shared` 可以随便依赖其他包。恰恰相反，它应该保持很低层，尽量不依赖其他 Vue 包，这样其他包才能放心依赖它。

## Socratic Check

如果 compiler 生成了 `PatchFlags.CLASS`，runtime 为什么可以少做很多 diff 判断？

如果 `normalizeClass` 只放在 `runtime-dom` 里，而 compiler 或 SSR 又需要同样规则，会出现什么维护风险？

## Quick Summary
- `@vue/shared` 是 Vue monorepo 的低层公共工具包，被多个核心包复用。
- 它不只是 utils，还包含 `PatchFlags`、`ShapeFlags` 这类 compiler/runtime 共享协议。
- 学会 `@vue/shared` 后，再读 `reactivity`、`runtime-core`、`compiler-core` 会顺很多。
