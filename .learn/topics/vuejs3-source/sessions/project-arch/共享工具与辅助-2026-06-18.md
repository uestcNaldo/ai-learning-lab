## Positioning

`isObject`、`hasChanged`、`hasOwn` 位于 `@vue/shared`，是 Vue 3 源码里最底层的一组判断工具，服务于响应式系统、运行时组件访问、props 处理、compiler 分析等多个包。

## Analogy

可以把它们想成仓库入口的三个小检查员：

- `isObject`：检查“这个东西能不能进入对象处理通道”。
- `hasChanged`：检查“新货和旧货是不是确实不同，需要触发后续流程吗”。
- `hasOwn`：检查“这个标签是不是贴在当前货物自己身上，而不是从别处继承来的”。

它们每个都很小，但因为站在入口处，所以一旦判断不统一，后面的响应式追踪、组件属性读取、更新触发都会出现细微偏差。

## Core Mechanism

### 1. `isObject`：判断值是否是可被对象逻辑处理的目标

Vue 源码中的实现很直接：

```ts
export const isObject = (val: unknown): val is Record<any, any> =>
  val !== null && typeof val === 'object'
```

它有两个重点：

- `null` 被排除，因为 `typeof null === 'object'` 是 JavaScript 历史遗留问题。
- 函数不算 object，因为这里的语义不是“广义对象”，而是“普通对象、数组、Map、Set 这类可以进入响应式代理或对象分支处理的值”。

在响应式系统里，`reactive()` 会先用 `isObject(target)` 判断目标能不能被代理。如果传入的是数字、字符串、布尔值，Vue 不会创建 Proxy，而是直接返回原值。

`baseHandlers.get` 里也会用 `isObject(res)` 判断读取出来的嵌套值是否需要懒转换成 `reactive(res)` 或 `readonly(res)`。这就是 Vue 深层响应式能够“按需展开”的关键之一。

### 2. `hasChanged`：判断值是否真的发生变化

Vue 源码中的实现是：

```ts
export const hasChanged = (value: any, oldValue: any): boolean =>
  !Object.is(value, oldValue)
```

它不是简单地写成：

```ts
value !== oldValue
```

原因是 `Object.is` 对一些 JavaScript 边界值更精确：

```ts
NaN !== NaN          // true
Object.is(NaN, NaN) // true

+0 === -0            // true
Object.is(+0, -0)   // false
```

所以 `hasChanged(NaN, NaN)` 会返回 `false`，避免无意义地触发更新。

在 `ref` 的 setter 中，Vue 会先比较新旧值：

```ts
if (hasChanged(newValue, oldValue)) {
  this._rawValue = newValue
  this._value = toReactive(newValue)
  this.dep.trigger()
}
```

这说明 `hasChanged` 是响应式更新的“闸门”：只有它认为值变了，后面才会触发依赖更新。

在 `reactive` 对象的 `set` handler 里，它也用于区分：

- 新增属性：触发 `ADD`
- 已有属性且值变化：触发 `SET`
- 已有属性但值没变：不触发

### 3. `hasOwn`：安全判断自有属性

Vue 源码中的实现是：

```ts
const hasOwnProperty = Object.prototype.hasOwnProperty

export const hasOwn = (
  val: object,
  key: string | symbol,
): key is keyof typeof val => hasOwnProperty.call(val, key)
```

它不直接写：

```ts
val.hasOwnProperty(key)
```

因为有些对象可能没有 `hasOwnProperty`，例如 `Object.create(null)` 创建的对象；也有些对象可能自己定义了一个同名属性，把原型方法覆盖掉。

它也不等同于：

```ts
key in val
```

因为 `in` 会检查原型链，而 `hasOwn` 只关心属性是否属于当前对象自己。

在 Vue 组件实例代理里，访问 `this.foo` 时会按顺序查找：

1. `setupState`
2. `data`
3. `props`
4. `ctx`

这些判断大量使用 `hasOwn`。如果用 `in`，原型链上的属性也可能被误判成组件自己的状态，组件访问缓存就不可靠了。

## Code Example

```ts
const hasOwnProperty = Object.prototype.hasOwnProperty

function isObject(val: unknown): val is Record<any, any> {
  return val !== null && typeof val === 'object'
}

function hasChanged(value: unknown, oldValue: unknown): boolean {
  return !Object.is(value, oldValue)
}

function hasOwn(val: object, key: string | symbol): boolean {
  return hasOwnProperty.call(val, key)
}

function maybeReactive(value: unknown) {
  if (!isObject(value)) {
    return value
  }

  return new Proxy(value, {
    get(target, key, receiver) {
      return Reflect.get(target, key, receiver)
    },
    set(target, key, newValue, receiver) {
      const oldValue = Reflect.get(target, key, receiver)
      const hadKey = hasOwn(target, key)
      const result = Reflect.set(target, key, newValue, receiver)

      if (!hadKey) {
        console.log('ADD', String(key), newValue)
      } else if (hasChanged(newValue, oldValue)) {
        console.log('SET', String(key), newValue, oldValue)
      } else {
        console.log('NO CHANGE', String(key))
      }

      return result
    },
  })
}

const state = maybeReactive({ count: 0 }) as { count: number; name?: string }

state.count = 1
state.count = 1
state.name = 'Vue'
```

这段代码展示了三个工具如何协作：

- `isObject` 决定传入值是否值得创建 Proxy。
- `hasOwn` 决定一次赋值是新增属性还是修改已有属性。
- `hasChanged` 决定已有属性的新旧值是否真的不同。

运行结果会接近：

```text
SET count 1 0
NO CHANGE count
ADD name Vue
```

这正是 Vue 响应式系统里的核心判断模型：先判断目标能不能被代理，再判断 key 原来有没有，最后判断值有没有变。

## Common Misconceptions

- 误区 1：`isObject` 等于“所有 JavaScript 对象”。在 Vue 这段源码里，它排除了 `null` 和函数，更关注可进入对象处理/响应式代理的值。
- 误区 2：`hasChanged` 可以简单替换成 `!==`。这样会错误处理 `NaN`，导致本不该触发的更新被触发。
- 误区 3：`hasOwn` 和 `key in obj` 一样。`in` 会查原型链，`hasOwn` 只检查当前对象自己的属性。
- 误区 4：这些函数太小，不值得学。恰好相反，它们是很多复杂模块的共同判断入口，统一它们可以减少全仓库的行为分叉。

## Socratic Check

1. 如果 `reactive(1)` 不用 `isObject` 先拦住，会发生什么？答案是：Vue 会尝试对非对象创建响应式代理，而 Proxy 只能代理对象，这会让响应式入口变得不安全。
2. 如果 `ref(NaN).value = NaN` 用 `!==` 判断变化，会发生什么？答案是：它会误以为值变化了，因为 `NaN !== NaN` 是 `true`，从而触发一次不必要的更新。

---

## Quick Summary

- `isObject` 是对象处理入口，决定值能不能进入响应式代理或对象分支逻辑。
- `hasChanged` 是更新触发闸门，用 `Object.is` 精确判断新旧值是否不同。
- `hasOwn` 是安全的自有属性检查，避免原型链和对象自身覆盖 `hasOwnProperty` 带来的误判。
